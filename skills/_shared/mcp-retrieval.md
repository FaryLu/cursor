# MCP 历史文档检索规则

适用于 `user-mzfileserver`。调用任何 MCP 工具前，必须先读取对应工具 descriptor，确认参数后再调用。

## 什么时候必须查 MCP
- 本地历史 PRD 未找到高相似度案例时，必须查 MCP 补充历史参考。
- 需求涉及跨版本复用、老功能增强、历史规则延续、跨端能力时，优先查 MCP。
- 本地文档能找到部分参考，但关键业务规则仍不清晰时，必须查 MCP 补充。

## 检索顺序
1. **关键词搜索**：按需求关键词、字段、业务动作搜索，优先查找最接近的历史需求标题、正文描述、业务规则（`search_by_keyword`）。
2. **模块搜索**：若已识别主功能模块，按模块名搜索对应历史需求（`search_by_module`）。
3. **影响范围搜索**：若已识别关键影响范围（如 `报表`、`客户`、`库存`、`权限`），按影响范围搜索（`search_by_impact_scope`）。
4. **结果择优**：若关键词结果很多，优先保留与当前需求最接近的版本、业务场景、字段和流程设计。
5. **失败回退**：若关键词、模块、影响范围三轮搜索后仍无足够结果，回退到本地历史 PRD + 模板推导，并触发追问机制。

## 详情读取
搜索结果中出现可疑相似需求时，使用 `get_requirement_detail` 读取详情后再归纳。

## 怎么使用 MCP 结果
- MCP 结果主要用于补充业务背景、影响模块、历史先例。
- 不能直接照抄 MCP 内容，必须结合当前需求、当前模板和本地历史 PRD 重新归纳。
- 若 MCP 中存在更接近当前需求的历史案例，可优先参考其业务规则和影响范围。
- 若不同检索入口返回多条历史记录，优先选择同时满足「功能接近 + 字段接近 + 业务流程接近」的结果。

## 冲突处理
- 若本地历史 PRD 与 MCP 历史记录冲突，优先采用更接近当前版本、当前业务形态、当前需求场景的内容。
- 若仍无法判断，以当前 `秒账新功能影响模块模板` 和当前需求描述为主，并将冲突点标记为 `待确认项`。

## 失败回退
- 若 MCP 无结果、结果过少、或结果无法支持判断，则回退到本地历史 PRD + 模板分析继续输出。
- 回退后若仍无法可靠判断核心业务规则，必须触发追问，不能强行补全。
