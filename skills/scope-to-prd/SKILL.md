# Scope To PRD

## 核心目标
本 skill 用于在用户已经提供确认后的 `scope` 时，直接生成最终 PRD 文档。

它不是三步式 agent，不需要先单独输出影响模块给用户确认，再继续生成。

本 skill 的工作方式是：
1. 先把用户给出的 `scope` 结构化
2. 再检索本地历史 PRD 和 MCP 历史记录补充业务规则
3. 最后直接生成 PRD 文档

## 适用场景
当出现以下任一情况时，优先使用本 skill：
- 用户已经给出确认后的影响模块，不需要再单独确认影响模块
- 用户会以 `模块 -> 页面 -> 区域` 的结构给出需求信息
- 用户的输入来自脑图，因此可能是 markdown 或图片
- 用户希望直接得到最终 PRD，而不是先走“分析影响模块 -> 再确认 -> 再生成”

## 不适用场景
以下场景不要单独使用本 skill：
- 用户只给了一句模糊需求，没有给 scope
- 用户还没有确认影响模块，希望先做影响范围分析
- 用户更需要“先讨论边界和影响模块”，而不是直接生成 PRD

## 输入要求
输入至少应包含以下之一：
- 用户提供的 scope markdown
- 用户提供的脑图截图、图片、导图导出的图片
- 用户提供的模块 / 页面 / 区域清单

scope 中最好包含：
- 需求名称
- 边界说明
- 模块列表
- 页面列表
- 区域列表
- 关键业务规则
- 明确不做的范围

## 输入优先级
生成时按以下优先级判断，不可颠倒：
1. 用户当前提供的 scope、边界说明、规则说明
2. 用户在当前对话中补充确认的信息
3. 最接近当前场景的本地历史 PRD
4. 最接近当前场景的 MCP 历史记录
5. `秒账新功能影响模块模板.md` 中的横切检查项和补漏规则

规则：
- 用户明确写入 scope 的内容优先级最高
- 历史文档用于补充规则、补充表达、补充常见设计，不用于推翻用户已确认 scope
- 若历史文档与用户确认 scope 冲突，以用户当前确认内容为准

## 图片和 Markdown 处理
### 1. Markdown scope
- 直接按标题、列表、缩进识别层级
- 优先识别：`边界说明`、`影响模块`、`模块 -> 页面 -> 区域`、`规则`

### 2. 图片 / 脑图输入
- 先把图片内容整理成结构化文本，再开始检索历史文档
- 保留层级关系，尽量还原为：`模块 -> 页面 -> 区域 -> 字段 / 规则`
- 若图片可读性不足、文字缺失、层级不清，必须先追问，不能强行猜测

## 历史文档检索范围
### 1. 本地搜索
- `../../data/prd/*/*.md`
- `../../data/modules/秒账新功能影响模块模板.md`

### 2. MCP 搜索
- MCP 中 `user-mzfileserver` 的历史需求记录

## 历史检索策略
### 第一步：先结构化 scope
从用户输入中提取：
- 主功能名称
- 业务对象
- 已确认影响模块
- 页面
- 区域
- 字段或规则
- 明确“不做”的范围
- 端能力信息，例如 PC / Pad / App / 云店

### 第二步：按 scope 单元拆分检索
不要只搜索需求标题，必须按 scope 中的关键单元分别检索：
- 模块名
- 页面名
- 区域名
- 关键字段
- 关键动作
- 关键规则

例如：
- 若 scope 中包含 `销售单 -> 详情页 -> 开票记录`
- 需要分别搜索：`销售单`、`开票`、`发票`、`抬头`、`详情页`、`打印`、`导出`、`权限`

### 第三步：多版本去重和择优
由于同一个功能可能出现在多个版本中，必须做版本聚合：
- 先把相同功能、相同主题、相同字段的历史文档视为同一类案例
- 优先选择与当前业务场景最接近的案例
- 在同样接近的前提下，优先使用较新版本
- 较老版本只能用于补充规则演进、兼容口径、历史做法，不能机械覆盖新版本结论

### 第四步：MCP 使用规则
出现以下任一情况时，必须查 MCP：
- 本地历史 PRD 找不到足够接近的案例
- 同一功能在多个版本中反复出现，需要补充历史演进线索
- 本地能找到模块，但关键字段、交互或业务规则仍不清晰

MCP 的用途：
- 补充历史先例
- 补充跨版本规则演变
- 补充同功能在其他版本中的字段与交互

### 第五步：冲突处理
- 若本地历史 PRD 与 MCP 冲突，优先使用更接近当前场景、当前版本、当前业务边界的内容
- 若历史内容彼此冲突，但用户 scope 已明确，以用户 scope 为准
- 若冲突点无法判断，必须在 PRD 中标记为 `待确认项`

## 生成原则
1. 以用户提供的 scope 为主，不重新做影响模块确认流程。
2. 可以补齐历史规则，但不能擅自扩写出用户已经明确排除的范围。
3. 对于 scope 中已经明确写出的模块、页面、区域，生成时必须尽量一一落到 PRD 结构里。
4. 对于历史文档中高频出现、但用户 scope 未明确排除的横切内容，允许补充：
   - 权限
   - 打印
   - 导入
   - 导出
   - 报表
   - 业务日志
   - 老数据兼容
   - 新账号默认值
5. 若用户 scope 已明确写“不做”，则这些内容不能强行写入最终需求范围，只能在 `待确认项` 或补充建议中提及。

## 必须追问条件
出现以下任一情况时，不能直接生成 PRD，必须先追问：
- scope 中只有模块，没有页面和区域，且无法从历史文档高置信补全
- scope 中有页面，但核心业务规则未说明
- 涉及状态流转、上下游单据、金额、库存、权限、打印、导入导出，但关键规则缺失
- 图片中的文字、层级、节点关系无法可靠识别
- 用户提供的 scope 与历史文档差异很大，且无法判断哪些是刻意变更

## PRD 生成要求
输出格式、层级结构、权限写法、数据规则写法，严格复用 `generate-final-prd` 的要求。

生成时必须满足：
- 保持 PRD 标题层级：`# ## ### ####`
- 一级内容使用编号列表 `1. 2. 3.`
- 下级内容使用无序列表 `-`
- 数据权限、字段权限、操作权限必须区分清楚
- 列表字段必须先判断基础字段、敏感字段、条件字段
- 选择器、下拉框、联想搜索必须写数据范围
- 对用户 scope 中已经明确的模块、页面、区域，不允许只写成一句概括

## 推荐工作流
1. 读取用户提供的 scope
2. 若是图片，先整理成结构化文本
3. 提取模块、页面、区域、规则关键词
4. 搜索本地历史 PRD
5. 若不足，再搜索 MCP
6. 聚合同类功能的多个版本结果，择优取用
7. 对照模板检查是否遗漏高频横切模块
8. 直接生成最终 PRD
