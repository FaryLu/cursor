# Scope To PRD Examples

## 示例一：Markdown scope 直接生成 PRD

### 输入

```markdown
# 销售单开发票 — 影响模块（已确认）

## 边界说明（产品确认）
- 不做：第三方税控平台对接、独立发票管理菜单
- 要做：销售单维度开票台账、多条开票记录
- 规则：允许同一销售单登记多条开票记录

## 影响模块
- 商户设置（业务属性、基本信息、偏好设置）
- 销售单（列表、详情/编辑、导入、导出、打印、权限）
- 客户（档案、列表、导入）
- 权限
- 员工管理 / 业务日志
```

### 处理方式
1. 先把 `边界说明` 和 `影响模块` 视为已确认 scope
2. 不再单独输出影响模块让用户二次确认
3. 直接检索历史 PRD：
   - 销售单
   - 开票 / 发票 / 抬头
   - 客户资料
   - 打印 / 导出 / 权限 / 日志
4. 若本地历史不足，再补充 MCP
5. 最终直接生成 PRD 文档

### 输出要求
- 输出 PRD，而不是影响模块分析
- 影响模块章节可直接复用用户已确认 scope
- 历史文档只用于补规则、补字段、补交互，不推翻用户边界

## 示例二：脑图图片生成 PRD

### 输入
- 用户给出脑图截图
- 脑图内容包含：
  - 模块：销售单、客户、打印、权限
  - 页面：列表、详情、编辑页
  - 区域：开票记录区、筛选区、打印区

### 处理方式
1. 先把图片整理成结构化文本：
   - `销售单 -> 列表页 -> 搜索区`
   - `销售单 -> 详情页 -> 开票记录区`
   - `客户 -> 详情页 -> 开票信息区`
2. 若图片文字识别不清、层级不清，先追问
3. 再按结构化后的文本检索历史 PRD 和 MCP
4. 直接生成最终 PRD

## 示例三：多版本历史文档处理

### 场景
- 同一个功能在多个版本中都有记录
- 本地历史文档和 MCP 中都能搜到

### 正确做法
1. 先把同主题结果聚合为同一类案例
2. 优先使用更接近当前场景的版本
3. 在场景同样接近时，优先使用更新版本
4. 旧版本只用于补充历史演进、兼容口径、旧规则
5. 若用户当前 scope 已明确，则以当前 scope 为准

## 示例四：哪些情况必须追问

### 需要追问
- 只有模块，没有页面和区域
- 图片内容无法可靠识别
- 关键业务规则缺失，例如是否允许编辑、是否实时生效、是否影响打印导出
- scope 与历史文档差异很大，无法判断哪些是刻意变更

### 不需要追问
- 用户已经明确写出边界说明、模块、页面、区域、规则
- 历史文档只是在字段命名或旧交互上有轻微差异
