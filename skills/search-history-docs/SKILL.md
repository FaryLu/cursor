## 前置上下文

执行本 skill 前，必须读取：@skills/_shared/prd-context.md

## 核心目标

不是只做关键词搜索，而是基于「需求描述 + 秒账核心功能 + 秒账功能关联关系 +（若涉及云仓）跨系统 L0/L2c + 历史 PRD/MCP」输出完整的影响模块分析，稳定识别新功能带来的关联功能影响。

分析顺序从大到小：**判定是否跨系统（L0）→ ERP 框架（L2a/L2b）→ 跨系统叠加（L2c）→ 历史细节（PRD/MCP）**。

## 资料范围

资料分两类，用法不同，不可混为一谈：

### 1. 结构化知识库（按章节读取，用于影响范围分析）

**ERP 系统内（始终执行）**
- `../../data/modules/秒账核心功能.md` — 全局能力校验清单（L2a）
- `../../data/modules/秒账功能关联关系.md` — 主功能关联展开与补漏规则（L2b）

**跨系统（命中 prd-context「L0/L2c 触发条件」时执行，与 L2a/L2b 叠加）**
- `../../data/modules/系统定位-三系统.md` — 主系统判定、货主生命周期、端能力矩阵
- `../../data/modules/_模块词典.md` — 模块规范名（输出须用 `[ERP]`/`[WMS]`/`[BSS]` 前缀）
- `../../data/modules/跨系统功能关联关系.md` — 跨系统关联与补漏（CS-01～CS-18）
- `../../data/modules/用户场景矩阵.md` — 可选；匹配场景 ID 辅助补漏
- `../../data/modules/_跨系统语料索引.md` — 下钻 MCP/本地 PRD 原文时用

**L3** — `../../data/modules/` 下独立文档，按核心功能、关联关系或 CS 证据引用按需读取

### 2. 历史文档检索（关键词搜索，用于补充子能力与业务规则）

- `../../data/prd/*/*.md` — 历史 PRD；跨系统优先读 `_跨系统语料索引.md` 中 P0/P1
- MCP — 按 @skills/_shared/mcp-retrieval.md：
  - `user-mzfileserver`：秒账 ERP
  - `user-wmsfileserver`：WMS + 运营/BSS（**跨系统需求必须查**）
- 检索词优先取自：需求描述、L2 模块名、`_模块词典.md` MCP 模块名、CS 节「证据」中的版本-序号

## 分析策略

### 关键词提取

从需求描述中提取，供后续 L2 匹配与历史检索使用：

1. **业务关键词**：采购、销售、库存、财务等
2. **功能关键词**：订单、报表、审批、导入等
3. **对象关键词**：商品、客户、供应商、仓库等
4. **关联关键词**：生成单据、打印、导入导出、权限、日志、提醒、兼容老数据等

### 执行步骤

按以下顺序执行，不能跳步：

0. **判定是否跨系统（L0）**
   - 对照 @skills/_shared/prd-context.md「L0/L2c 触发条件」
   - 若命中：读取 `系统定位-三系统.md` §8 checklist，标注**主系统**（ERP / WMS / BSS）及是否涉及货主未审批、账期、最后一票货等
   - 若不命中：跳过步骤 3b、5 中 WMS/BSS 专项与输出中的「跨系统影响」节，仅走 ERP L2

1. **识别主功能**
   - 提炼主功能、核心对象、核心业务动作、上下游单据/模块
   - 对照 `秒账核心功能.md` 第 1 节，确定最接近的主功能模块
   - 跨系统时：用 `_模块词典.md` 将口语（云仓出库、运营平台月账等）对齐为规范名
   - 明确属于「新增功能 / 规则调整 / 字段扩展 / 流程变更 / 兼容性修改」中的哪一类

2. **对照核心功能清单初筛（L2a）**
   - 读取 `秒账核心功能.md`，用「触发条件」匹配需求可能涉及的能力
   - 结论：✅影响 / ⬜不影响 / ❓待确认；禁止把清单条目直接当作已分类的三层结果
   - 同步关注第 0 节全局必检项（字段权限、老数据兼容、新账号默认值）

3. **从关联关系展开候选影响项（L2b + L2c）**

   **3a. ERP 系统内（L2b，始终执行）**
   - 读取 `秒账功能关联关系.md` 中主功能对应章节
   - 同时读取：全局补漏规则、生成单据/消息提醒/通用能力/开单方式/重要属性等横向章节（按需求命中情况）
   - 若全局补漏提示涉及 WMS/云仓 → 必须进入 3b

   **3b. 跨系统（L2c，仅 L0 命中时执行）**
   - 读取 `跨系统功能关联关系.md` §1.2 触发索引 + §2 全局补漏 + 命中 CS 章节（CS-01～CS-18）
   - 与 `秒账功能关联关系.md` **合并去重**；模块名一律用 `_模块词典.md` 规范名
   - 可选：对照 `用户场景矩阵.md` 匹配场景 ID，核对是否遗漏 CS

   - 合并 3a/3b 与第 2 步初筛结果，不得因用户未直接提到而省略明显相关项

4. **分层判断影响范围（输出归类）**
   - **直接影响**：需求主流程直接发生修改的模块
   - **间接影响**：由主流程联动、生成、引用、同步、回写、反查、状态变化导致受影响的模块
   - **横切影响**：权限、导入、导出、打印、报表、消息提醒、业务日志、老数据兼容、新账号默认值、自定义模板等共性能力
   - 每个影响模块必须说明「为什么会被影响」，并标注来源：关联关系 / 跨系统关联(CS-xx) / 核心功能校验 / 业务规则推导
   - **跨系统模块**须标注系统前缀（如 `[WMS]出库单`、`[BSS]月账单`）
   - 子能力列不全或关键规则不清晰时，进入第 5 步补充；暂无法判断的标注 `待确认项`

5. **检索历史依据（补充细节）**
   - 针对第 4 步已识别的模块/子能力，在历史 PRD 中搜索同类功能、相似单据、相同字段、相同交互、相同业务规则
   - **跨系统**：先查 `_跨系统语料索引.md` 对应主题 → `get_requirement_detail(version, index)`；本地 P0 如 `V4.9.0`、`V5.4.0`
   - 本地不足时，按 MCP 规则补充搜索（优先 `search_by_module`、`search_by_impact_scope`）
   - 跨系统需求：**必须**检索 `user-wmsfileserver`；秒账侧用 `user-mzfileserver` 或本地 PRD
   - 历史结果用于：补充子模块名称、验证关联合理性、提供业务规则先例；不得用历史案例推翻 L2/L2c 框架分析
   - 补充完成后，更新各影响项的来源标注，追加：本地历史 PRD 复用 / MCP 历史记录复用

6. **输出前补漏自检**
   - 对照 `秒账功能关联关系.md` 全局补漏规则与主功能补漏规则
   - **跨系统**：对照 `跨系统功能关联关系.md` §2 全局补漏 + 已读 CS 节是否覆盖需求触发点
   - 对照 `秒账核心功能.md` 第 0 节全局必检项
   - 若需求涉及单据、字段、状态、金额、库存、扫码、模板、生成下游单据，必须重点检查
   - 跨系统额外检查：货主是否审批通过、账期/月账单、WMS 作业状态回传、三端展示是否一致
   - 若关联关系或核心功能写有「可省略」说明，只有满足条件且能说明原因时，才允许不纳入输出
   - 若本地历史 PRD 和 MCP 都无法支撑关键判断，停止输出最终结论，改为先追问

### 历史检索优先级

适用于第 5 步，按优先级依次尝试：

1. 精确匹配（关键词完整出现）
2. 模糊匹配（关键词部分匹配）
3. 语义匹配（相关概念扩展）

## 必须追问条件

出现以下任一情况时，不能直接输出影响模块结论，必须先追问：

- 无法识别需求最接近哪个主功能模块。
- 无法判断用户角色、核心使用场景、业务目标。
- **跨系统**：无法判断主系统（ERP/WMS/BSS），或涉及云仓但未说明是否已货主签约/审批通过。
- **跨系统**：涉及出库/账单/账期，但无法判断是否已过账期、是否最后一票货、月账单是否需重算。
- 涉及上下游单据联动，但无法判断是否生成下游单据、是否回写来源单据、是否影响状态流转。
- 涉及字段、金额、数量、库存、价格、状态、审核，但关键规则不清晰（如取值口径、是否允许编辑、是否允许负数、是否需要兼容老数据）。
- 需求明显可能影响权限、打印、导入、导出、报表、消息提醒、业务日志，但用户未说明是否需要同步调整，且无法从关联关系、核心功能和历史文档中明确判断。
- 需求涉及商户设置开关、新账号默认值、老数据兼容，但未说明开启关闭后的处理方式。
- 关联关系或核心功能提示应检查多个模块，但无法判断其中哪些需要纳入。
- 本地历史 PRD 与 MCP 结果都不足以支撑关键业务判断。

追问后再生成：只有在核心目标、主功能模块、关键业务规则、上下游关系基本明确后，才能输出影响模块；部分细节仍不明确但可高置信推断时，须标注 `待确认项`。

## 输出格式

```markdown
## 影响模块

> 跨系统需求时，模块名须带 `[ERP]`/`[WMS]`/`[BSS]` 前缀（见 `_模块词典.md`）。

### 〇、跨系统影响（仅 L0 命中时输出；纯 ERP 需求删除本节）
- **主系统**：[ERP / WMS / BSS] — 判定依据（一句话）
- **关联 CS**：CS-xx、CS-yy
- **模块名称**
  - 子模块：摘要
  - 关联理由：为什么受影响
  - 来源：跨系统关联(CS-xx) / MCP / 本地 PRD

### 一、直接影响
- **模块名称**
  - 子模块：摘要
  - 关联理由：为什么受影响
  - 来源：关联关系 / 跨系统关联(CS-xx) / 核心功能校验 / 本地历史PRD复用 / MCP历史记录复用 / 业务规则推导

### 二、间接影响
- **模块名称**
  - 子模块：摘要
  - 关联理由：为什么受影响
  - 来源：关联关系 / 核心功能校验 / 本地历史PRD复用 / MCP历史记录复用 / 业务规则推导

### 三、横切影响
- **模块名称**
  - 子模块：摘要
  - 关联理由：为什么受影响
  - 来源：关联关系 / 核心功能校验 / 本地历史PRD复用 / MCP历史记录复用 / 业务规则推导

### 四、补漏检查
- 已检查：全局必检项、全局补漏规则、主功能关联表、核心功能清单相关分组
- 跨系统已检查：（若有）跨系统全局补漏、CS-xx 章节、货主生命周期/账期/三端
- 可省略项：写明省略原因；不能只省略不解释

### 五、核心功能校验记录
| 功能名称 | 结论 | 理由 | 输出归类 |
|----------|------|------|----------|
| [功能名] | ✅影响 / ⬜不影响 / ❓待确认 | [简述] | 直接/间接/横切/— |
```

## 输出文件

按 @skills/_shared/output-convention.md 写入 `output/scope/{需求名称}-影响模块.md`。

## 使用示例

**输入**：产品维度条形码功能

**输出**：

```markdown
## 影响模块
### 一、直接影响
- **商户设置**
  - 子模块：货品属性、打印信息
  - 关联理由：需要新增维度条形码开关与打印字段配置
  - 来源：关联关系 + 业务规则推导

- **产品**
  - 子模块：产品详情、导入
  - 关联理由：条形码字段要在产品基础资料中维护，并支持批量导入
  - 来源：关联关系 + 本地历史PRD复用

- **销售单**
  - 子模块：产品列表、扫码开单
  - 关联理由：销售开单需要识别维度条形码并正确带出产品
  - 来源：关联关系 + 本地历史PRD复用

### 二、间接影响
- **销售单**
  - 子模块：打印、导入
  - 关联理由：新增字段后，打印模板和导入模板需同步支持
  - 来源：关联关系

### 三、横切影响
- **打印**
  - 子模块：A4打印、热敏打印
  - 关联理由：涉及新增条形码字段时，常见打印能力需要同步评估
  - 来源：关联关系 + 核心功能校验

### 四、补漏检查
- 已检查：全局必检项、全局补漏规则、销售单/产品/商户设置关联、核心功能清单
- 可省略项：报表、消息提醒、业务日志暂无直接影响，可暂不展开

### 五、核心功能校验记录
| 功能名称 | 结论 | 理由 | 输出归类 |
|----------|------|------|----------|
| 打印 | ✅影响 | 新增条形码字段需进打印模板 | 横切影响 |
| 导入 | ✅影响 | 产品维度条形码需批量维护 | 横切影响 |
| 销售单生采购单 | ⬜不影响 | 本次不改销售采购联动 | — |
```
