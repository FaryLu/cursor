## 前置上下文

执行本 skill 前，必须读取：@skills/_shared/prd-context.md

## 文档档位

本 skill 产出 **标准骨架 PRD**（见全局规则文档档位定义）。优先保证模块/页面/区域结构完整和关键规则不遗漏，不追求全量字段级展开；全量细化交由 `refine-prd-detail`。

# 生成最终PRD

## 功能概述

按公司标准模板格式，整合前置分析结果生成标准结构 PRD 草稿。

本 skill 重点保证：
- 需求背景清楚
- 影响模块完整
- 页面结构完整
- 关键规则不遗漏

本 skill 默认不追求把每个页面字段都细化到最高颗粒度。
当用户明确要求字段级、按钮级、权限级、数据范围级展开时，应在本 skill 产出骨架后，再交给 `refine-prd-detail` 继续细化。

## 输入依赖
- 用户已确认的影响模块（`search-history-docs` 产出）
- 用户原始需求描述
- `秒账功能关联关系.md` 与 `秒账核心功能.md`（用于生成前补漏复核）

## 与第 1 步的分工

| 阶段 | 粒度 | 职责 |
|------|------|------|
| `search-history-docs` | 模块 / 子模块 | 判断「哪些模块受影响」 |
| 本 skill | 页面 / 区域 / 关键字段 / 按钮 | 判断「各模块怎么写进 PRD」 |

第 1 步的影响模块不足以直接写出页面结构；本 skill **必须**基于已确认影响模块，再次深入检索本地历史 PRD 和 MCP，补齐各模块的页面、区域、字段、按钮与业务规则。

## 历史文档检索

检索范围与规则见 @skills/_shared/mcp-retrieval.md；本地路径：`../../data/prd/*/*.md`。

### 检索触发

梳理完影响模块后、设计功能模块前，对每个影响模块（含子模块）分别检索，不得只依赖第 1 步摘要。

### 检索单元

按「模块 → 页面 → 区域 → 字段/按钮/规则」拆分关键词，分别搜索：

- 模块名、子模块名（如 `销售单`、`扫码开单`、`打印`）
- 页面名（如 `列表页`、`新建页`、`详情页`）
- 区域名（如 `搜索区`、`产品明细`、`操作列`）
- 本次新增或修改的关键字段、按钮、业务动作
- 横切能力（`权限`、`导入`、`导出`、`打印`、`报表`、`业务日志`）

示例：影响模块含 `销售单 -> 产品列表、扫码开单`，需分别检索 `销售单`、`产品列表`、`扫码开单`、`条码`、`维度条码` 及关联的 `打印`、`权限`。

### 检索顺序

1. 本地历史 PRD：优先找同模块、同页面、同字段的高相似度案例
2. MCP：本地不足，或关键字段/交互/规则仍不清晰时，按 MCP 规则补充（优先 `search_by_module`、`search_by_impact_scope`）
3. L3 模块文档：关联关系或核心功能引用了独立规则文档时按需读取

### 结果使用

- 用于确定：页面有哪些、区域怎么分、现有字段/按钮/权限/数据规则如何复用
- 不能直接照抄历史 PRD，须结合当前需求和已确认影响模块重新归纳
- 多版本结果择优：功能接近 + 字段接近 + 业务流程接近；较新版本优先，老版本仅补充兼容口径
- 写「复用 XX 模块 XX 规则」时，须能对应到具体检索来源（历史 PRD 章节或 MCP 记录）

## 生成步骤

1. **确定需求名称**
   - 基于用户原始需求提炼简洁明了的需求名称

2. **编写需求背景**
   - 整合第 1 步影响模块分析及本 skill 历史检索中的业务背景
   - 说明本次需求要解决的问题
   - 补充相关上下文信息

3. **生成前补漏复核**
   - 不能机械照抄 `search-history-docs` 的结果
   - 必须再次对照 `秒账功能关联关系.md` 全局补漏规则与主功能章节、`秒账核心功能.md` 全局必检项
   - 重点复核：权限、打印、导入、导出、报表、消息提醒、业务日志、老数据兼容、新账号默认值、自定义模板
   - 若发现遗漏但能够明确判断受影响，先标注遗漏项并反馈用户确认；用户已确认影响模块后，不得擅自扩大范围，可写入 `待确认项`

4. **梳理影响模块**
   - 基于用户已确认的影响模块分析结果，列出所有受影响的模块
   - 每个模块后括号内列出具体子模块
   - 优先覆盖直接影响模块，其次补齐间接影响与横切影响模块
   - 不得擅自扩大用户已确认的影响范围

5. **按影响模块深入检索**
   - 按「历史文档检索」章节，对每个影响模块及其子模块分别查本地 PRD 和 MCP
   - 目标：补齐各模块的页面清单、区域划分、现有字段/按钮/权限/数据规则
   - 检索完成后，形成「模块 → 页面 → 区域 → 可复用规则」的结构化笔记，供后续步骤使用
   - 若某模块/页面无法从历史文档高置信补全，标注 `待确认项` 或触发追问，不得凭空编造页面结构

6. **设计功能模块**
   - 基于第 5 步检索结果，按模块逐个展开功能设计
   - 每个模块下分功能页面
   - 每个页面下分页面区域
   - 优先产出结构完整、可继续细化的页面骨架，不必在本步骤强行把所有字段写到最高颗粒度

7. **定义字段与交互**
   - 列出页面关键字段和关键按钮，保证主流程、关键判断条件、核心交互完整
   - 对新增关键字段说明取值、输入限制（输入格式、长度、类型、必填）、展示或交互逻辑
   - 对关键按钮说明用途、触发动作、核心前置条件
   - 对关键权限点进行结构化区分：
     - 字段权限：决定字段是否展示、是否只读
     - 操作权限：决定按钮和操作是否可用
     - 数据权限：写在“数据规则 / 数据范围 / 结果集范围”中
   - 对下拉框、选择器、联想搜索等关键选择型控件，需要说明候选数据受哪些范围限制
   - 字段/按钮/权限规则须基于第 5 步检索结果；不得无依据写「复用现有逻辑」
   - 若某页面需要逐字段、逐按钮、逐数据源展开到高颗粒度，交由 `refine-prd-detail` 继续细化

8. **编写数据规则**
   - 在对应区域下编写数据规则
   - 包括排序、筛选、权限、校验等
   - 对列表、详情、弹框、下拉数据源，都要说明数据可见范围和数据权限规则
   - 对搜索项、筛选项、下拉项，要说明「看得见什么数据」「能选中什么数据」「无权限时如何表现」
   - 必须避免把「数据权限」和「字段权限」混写成一句模糊描述
   - 数据权限重点写在：列表结果、详情数据可见范围、报表取值范围、下拉数据源、选择器候选范围、搜索结果范围

9. **编写验收标准**
   - 在 PRD 头部「3. 验收标准」章节输出
   - 至少覆盖 P0 主流程验收项，每条含前置条件、操作步骤、预期结果
   - P1/P2 按需求复杂度选填

## PRD标准模板结构

```markdown
# [需求名称]

1. 需求背景：
    - [背景说明]
    - [补充说明]
2. 影响模块：
    - [模块一名称]（[子模块1]、[子模块2]、[子模块3]）
    - [模块二名称]（[子模块1]、[子模块2]）
3. 验收标准：
    - P0：[主流程必须通过的验收项，每条含前置条件、操作步骤、预期结果]
    - P1：[重要但非阻塞项，可选]
    - P2：[优化项，可选]

## [同级别标题编号] [模块名称，如：销售单]

### [同级别标题编号] [页面名称，如：销售单列表页]

#### [同级别标题编号] [区域名称，如：列表区域]

1. [字段名/按钮名称，如：合同金额]：
    - [交互逻辑]：根据字段类型和所在位置考虑：展示逻辑、输入逻辑（格式、长度、类型、必填）、取值逻辑、交互逻辑
    - [字段权限]：字段项必须说明；无权限时说明表现（隐藏 / 只读 / 置灰 / 报错）
    - [操作权限]：按钮/操作项必须说明；无权限时说明表现
    - [数据范围]：仅当该项是下拉框、选择器、联想搜索、搜索结果、列表结果、报表取值时才需要说明可选/可见数据范围和受哪些数据权限影响
    - 举例：字段「合同金额」写交互逻辑 + 字段权限；按钮「保存」写交互逻辑 + 操作权限；下拉「仓库」额外写数据范围

2. [数据规则，根据区域性质，输出技术需要关注的不体现在ui层面的逻辑，如：列表页的数据规则]
    - 列表排序规则：创建时间倒序
    - 列表数据权限规则：有查看销售单权限显示，只能查看有查看权限的销售单
    - 列表分页数据规则：20条/页
```

## 生成PRD格式注意事项
- 严格遵循模板格式，保持层级结构（# ## ### #### #####）
- 需求背景控制在3-4点，简明扼要
- 影响模块列出具体子模块，用顿号分隔
- 如果某个横切模块经过复核确认无影响，可以不在最终PRD展开，但应确保已经完成过检查
- 功能描述使用编号列表（1. 2. 3.），编号分级
- 下级内容使用无序列表（- ）
- 数据规则放在对应页面区域下，使用五级标题
- 标准骨架稿优先覆盖关键字段、关键按钮、关键数据源及其权限和数据范围；不要求每个页面全量逐字段展开
- 若用户要求全量字段级细化，在本 skill 完成后调用 `refine-prd-detail`
- 不允许只写“权限同现有逻辑”或“数据范围同现有逻辑”而不指明复用来源；复用来源应来自本 skill 第 5 步的历史检索
- 不允许把“查看哪些数据”和“查看哪些字段”混为一谈；必须明确区分数据权限与字段权限
- 不允许把数据权限机械套到每个字段和按钮上；普通字段重点写字段权限，普通按钮重点写操作权限，数据权限放在数据规则和数据范围处描述
- 不允许把所有列表字段一刀切地写成“受字段权限控制”；必须先判断哪些是基础字段，哪些是敏感字段，哪些是条件字段

## 输出示例
**输入**：销售订单功能

**输出**：
```markdown
# 销售订单

1. 需求背景：
    - 销售订单用于前置承接销售计划、报价、客户确认等场景。
    - 用户可先录入订单，再根据订单生成或关联后续销售单，便于跟踪订单执行情况。
    - 本次需求重点是补齐销售订单的标准业务骨架，并考虑与销售单、客户、产品、报表、打印、导入导出、权限等模块的联动。
2. 影响模块：
    - 秒账：商户设置、销售订单、销售单、客户、产品、报表、打印、导入、导出、权限、业务日志
3. 验收标准：
    - P0：销售订单列表可正常查看、新建、编辑、删除；关联销售单后订单状态与数量正确回写。
    - P0：权限控制与现有销售单同类规则一致，无权限账号不可查看敏感金额字段。
    - P1：导入、导出、打印模板字段与订单字段一致。

## 1 销售订单
### 1.1 列表页
#### 1.1 搜索区
1. 分店、客户、单号、订单日期、订单状态：
    - 交互逻辑：支持按常用条件筛选销售订单；其中分店、订单状态为下拉筛选，客户、单号支持模糊匹配。
    - 字段权限：有查看销售订单列表权限时显示搜索区；其中客户、分店等字段若现有系统已有字段权限控制，则复用现有规则。
    - 数据范围：搜索结果仅返回当前账号有权查看的销售订单数据；下拉项仅展示当前账号有权查看和可选的数据范围。

#### 1.2 列表字段
1. 基础字段：单号、客户名称、订单日期、订单状态、制单人：
    - 交互逻辑：用于识别和定位销售订单，默认在列表展示。
    - 字段权限：基础字段默认固定展示；若现有系统对个别字段已有单独权限控制，则复用现有规则。
2. 敏感字段：订单金额、销售金额、已收款：
    - 交互逻辑：展示订单执行和金额信息，用于判断订单金额与后续销售进度。
    - 字段权限：受金额或价格类字段权限控制，无权限时隐藏或空值展示。
3. 操作列：
    - 交互逻辑：支持查看详情、编辑、删除、跳转关联销售单等操作。
    - 操作权限：按查看、编辑、删除、关联单据查看等现有权限规则控制显示与可点击性。

#### 1.3 列表数据规则
1. 排序规则：默认按订单日期倒序，相同日期按创建时间倒序；如支持列头排序，复用现有单据列表排序逻辑。
2. 数据权限：仅展示当前账号有权查看的销售订单；若权限为“查看自己新建”，仅返回本人数据；若受分店、客户、业务员等范围限制，按现有单据数据权限过滤。
3. 分页规则：默认 20 条/页。

#### 1.4 列表功能
1. 新建销售订单：
    - 操作权限：有新建权限时显示。
    - 交互逻辑：点击进入新建销售订单页。
2. 导入 / 导出 / 打印 / 批量删除：
    - 操作权限：按各自操作权限控制显示。
    - 交互逻辑：导入校验、导出范围、打印限制、批量删除条件复用现有销售单相关规则，并结合销售订单字段差异调整。

### 1.2 新建页 / 编辑页
#### 1.1 基本信息区
1. 客户、订单日期、账期、业务员、备注：
    - 交互逻辑：用于录入订单主体信息，其中客户、日期为核心字段，业务员是否显示受商户设置影响。
    - 字段权限：字段可见性和可编辑性复用现有销售单或销售订单权限规则；若已有业务员、客户等字段权限，则按现有逻辑执行。
2. 产品明细：
    - 交互逻辑：录入产品、数量、单价、金额、规格等订单明细信息；产品维度字段是否出现受商户设置和产品模式影响。
    - 字段权限：金额、单价等敏感字段受价格类字段权限控制；产品选择器的数据范围受产品、仓库、分店等现有数据权限影响。
3. 页面按钮：
    - 操作权限：保存、删除、取消、复制新增等按钮按新建或编辑权限控制。
    - 交互逻辑：保存前需校验主体信息和产品明细必填项，校验文案复用现有单据规则。

### 1.3 详情页
#### 1.1 主体信息区
1. 订单基本信息：
    - 交互逻辑：展示客户、日期、状态、业务员、备注等订单主体信息。
    - 字段权限：基础信息默认展示，敏感字段按字段权限控制。
2. 产品明细区：
    - 交互逻辑：展示产品、数量、单价、金额、销售进度等信息。
    - 字段权限：金额、单价、已收款等敏感字段按现有金额权限规则控制。
3. 页面按钮：
    - 操作权限：编辑、删除、打印、导出、生成销售单等按钮按现有权限逻辑控制。

## 2 关联模块
### 2.1 销售单
1. 销售订单可作为销售单的来源单据或关联单据。
2. 订单状态、销售数量、销售金额等信息需与销售单执行结果联动更新。

### 2.2 客户 / 产品
1. 客户资料影响客户选择、客户展示、联系方式等基础信息带出。
2. 产品资料影响产品选择、规格型号、颜色、单价、单位等明细信息展示与校验。

### 2.3 报表 / 打印 / 导入导出 / 权限 / 业务日志
1. 报表需补充销售订单相关统计口径。
2. 打印模板、导入模板、导出字段需补充销售订单对应字段。
3. 权限需覆盖列表查看、新建、编辑、删除、导入、导出、打印及敏感金额字段查看。
4. 业务日志需记录关键操作和关键字段变更。
```

按 @skills/_shared/output-convention.md 写入 `output/prd/{需求名称}.md`。
