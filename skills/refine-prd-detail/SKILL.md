# Refine PRD Detail

## 作用定位
本 skill 用于把已经成型的 PRD 草稿进一步细化成“可评审、可开发拆解”的版本。

它是后处理 skill，不负责重新做以下事情：
- 不重新判断需求背景是否成立
- 不重新做主功能识别
- 不重新完整产出影响模块分析
- 不重写整篇 PRD 的一级结构

它主要负责：
- 细化页面结构：搜索区、列表区、详情区、新建页、编辑页、弹框
- 细化字段说明：取值、格式、长度、必填、默认值、联动、校验
- 细化按钮说明：显示条件、操作权限、点击逻辑、前置校验、成功失败反馈
- 细化权限说明：字段权限、操作权限、数据范围
- 纠正“权限写法过于机械”或“字段颗粒度不一致”的问题

## 什么时候使用
出现以下任一情况时，优先使用本 skill：
- 用户明确要求“字段级”“按钮级”“权限级”展开 PRD
- 用户认为现有 PRD 太粗，只覆盖了模块级、页面级，没有细到字段
- 用户强调查看页、新建页、编辑页里的每个字段都要考虑权限
- 用户强调下拉框、选择器、联想搜索的数据范围要写清楚
- 用户指出列表页不能把所有字段都机械写成权限字段，需要区分基础字段和敏感字段

## 不要使用的场景
以下场景不要单独使用本 skill：
- 还没有明确需求背景、核心流程、主功能模块
- 还没有产出影响模块，或影响模块明显不完整
- 用户只想快速得到一版标准 PRD，不需要字段级细化

## 输入要求
输入至少应包含以下之一：
- 已生成的 PRD 草稿
- 已确认的影响模块
- 用户补充的关键业务规则

若以下信息缺失且无法高置信推断，必须先追问：
- 页面范围不清楚，例如是否包含列表、详情、新建、编辑、弹框
- 单据状态流转不清楚，例如是否实时生效、是否审核、是否允许删除回滚
- 金额、数量、库存、均价等关键字段规则不清楚
- 下拉数据源受哪些数据权限控制不清楚
- 哪些字段属于敏感字段、是否已有现成权限规则可复用不清楚

## 必须追问条件

与「输入要求」中「若以下信息缺失且无法高置信推断，必须先追问」同义；Agent 第 3 步执行本 skill 时以此为准。

## 工作原则
1. 先继承已有 PRD 结构，再细化，不要整篇重写。
2. 先按页面和区域拆解，再写字段和按钮，不要直接散写字段。
3. 优先复用现有模块的权限口径、文案口径、报错口径。
4. 能明确复用来源时，必须写清楚“复用哪个模块、哪个字段、哪个按钮规则”。
5. 不能为了显得完整，把同一句数据权限机械复制到每个字段上。

## 细化流程

### 第一步：确认细化范围
先判断本次要细化哪些页面和区域：
- 搜索区
- 列表区
- 列表功能区
- 新建页
- 编辑页
- 详情页
- 弹框 / 选择器 / 辅助页
- 报表 / 导出 / 打印 / 日志等横切区域

若用户只要求细化部分区域，只输出该部分，不强行铺满整篇。

### 第二步：先分类型，再写权限
对每个页面元素先分类，再决定写法：

#### 字段分类
- 基础字段：用户识别和使用页面必需的核心字段，如单号、日期、状态、名称
- 敏感字段：金额、成本、均价、售价、毛利、利润、电话、地址等
- 条件字段：受商户设置、业务模式、模块开关、产品维度控制的字段
- 计算字段：系统根据规则自动计算或回显的字段
- 选择型字段：下拉框、选择器、联想搜索、弹框选择字段

#### 操作分类
- 页面操作：新建、编辑、删除、导入、导出、打印、审核、反审核
- 行内操作：查看详情、编辑当前行、删除当前行、跳转关联单据
- 辅助操作：新增明细行、删除明细行、复制、清空、展开收起

### 第三步：按区域选择颗粒度

#### 搜索区
默认写法：
- 交互逻辑：输入框 / 下拉框 / 日期范围 / 多选
- 字段权限：该搜索项是否显示
- 数据范围：搜索后结果受哪些数据权限影响；若是下拉搜索项，还要写候选数据范围

#### 列表区
默认先判断字段是不是基础字段：
- 若是基础字段，默认固定展示，除非系统已有明确字段权限控制
- 若是敏感字段，优先写字段权限
- 若是条件字段，写清楚出现条件
- 若是跳转字段，补充点击后的详情权限或跳转权限

列表区重点补充：
- 列表结果的数据权限
- 排序规则
- 分页规则
- 敏感列导出规则

#### 新建页 / 编辑页
这是最重颗粒度区域。每个字段尽量写：
- 字段名称
- 是否必填
- 输入格式
- 长度限制
- 默认值
- 取值来源
- 字段权限
- 编辑权限
- 联动逻辑
- 报错提示
- 数据范围（仅选择型字段需要）

每个按钮尽量写：
- 按钮名称
- 显示条件
- 操作权限
- 点击逻辑
- 前置校验
- 成功提示
- 失败提示

#### 详情页
重点说明：
- 展示逻辑
- 取值来源
- 字段权限
- 是否展示快照值还是实时值
- 页面按钮的操作权限

#### 下拉框 / 选择器 / 联想搜索
必须补充：
- 候选数据来自哪里
- 受哪些数据权限影响
- 是否受仓库、分店、客户、供应商、员工、角色、业务范围过滤
- 无可选数据时如何表现

## 权限写法规则

### 1. 字段权限
字段权限只回答这些问题：
- 字段是否显示
- 字段显示后是只读还是可编辑
- 无权限时是隐藏、只读、置灰还是报错

### 2. 操作权限
操作权限只回答这些问题：
- 按钮或操作是否显示
- 是否允许点击
- 点击后无权限如何处理

### 3. 数据权限
数据权限只写在这些位置：
- 列表结果集
- 报表结果集
- 详情数据是否可见
- 下拉候选数据
- 选择器候选数据
- 联想搜索结果
- 导出范围

禁止出现以下写法：
- 把“只能看自己的单据”重复写进每个普通展示字段
- 把“字段可见”与“数据可见”混成一句模糊描述
- 按钮同时写字段权限和操作权限

## 基础字段判断规则
列表页里不要机械把所有字段都写成权限字段，必须先判断哪些属于基础字段。

通常可优先视为基础字段的例子：
- 单号
- 日期
- 状态
- 单据类型
- 制单人
- 基础名称类识别字段

但以下情况不能直接按基础字段处理：
- 字段本身是金额、成本、均价、毛利等敏感信息
- 字段受商户设置或业务模式控制
- 字段是跨模块跳转入口，涉及额外权限
- 现有系统本来就对该字段有明确字段权限控制

## 输出要求
输出时应尽量保持用户已有 PRD 的标题结构，只补充和改写必要部分。

完整细化示例见 [examples.md](examples.md)。

格式必须遵守以下规则：
- 字段、按钮、数据规则项必须使用编号列表 `1. 2. 3.`，不能省略编号。
- 每个字段或按钮下的属性必须逐行使用无序列表 `-` 展开，不能把多个属性压缩成一句长句。
- 当某个字段适用完整模板时，必须按模板顺序逐项输出；不要随意省略顺序。
- `交互逻辑`、`字段权限`、`操作权限`、`数据范围`、`联动逻辑`、`报错提示` 等内容应各占一行。
- 若某字段没有某项内容，可省略该项；但不能因为想压缩篇幅，把多个属性合并到同一行。

优先采用以下表达方式：
- 普通字段：`交互逻辑 + 字段权限`
- 选择型字段：`交互逻辑 + 字段权限 + 数据范围`
- 按钮：`操作权限 + 点击逻辑`
- 数据区域：单独集中写 `数据规则`

若需求已明确要求高颗粒度，可在新建页、编辑页、关键详情区使用更完整模板：

```markdown
1. 字段名称：
   - 是否必填：
   - 输入格式：
   - 长度限制：
   - 默认值：
   - 取值来源：
   - 字段权限：
   - 编辑权限：
   - 联动逻辑：
   - 报错提示：
   - 数据范围：
```

## 结果校验
输出前必须自检：
- 字段、按钮、数据规则项是否都使用了编号列表
- 字段下的属性是否都使用了无序列表逐行展开
- 是否误把数据权限写进每个普通字段
- 是否误把列表所有字段都写成字段权限控制
- 是否遗漏按钮的操作权限
- 是否遗漏下拉框、选择器、联想搜索的数据范围
- 是否遗漏详情页只读字段的权限表现
- 是否明确复用了现有模块规则，而不是只写“同现有”
- 验收标准是否覆盖关键 P0 场景，且可执行、可验证

## 简短示例

### 不推荐
- `合同金额：有权限可看；数据权限同现有。`

问题：
- 字段权限和数据权限混写
- 没说明无权限时怎么表现

### 推荐
- `合同金额：`
  - `交互逻辑：展示订单中的合同金额。`
  - `字段权限：有查看合同金额权限时显示，否则隐藏或空值，复用销售单金额字段权限逻辑。`

### 选择器推荐写法
- `仓库：`
  - `交互逻辑：下拉单选。`
  - `字段权限：有新建单据权限时显示。`
  - `数据范围：仅展示当前账号有权查看和可操作的仓库。`
