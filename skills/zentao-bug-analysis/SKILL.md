# 禅道 Bug 合理性分析

## 适用场景

当用户希望你：

- 分析某个或某批禅道 Bug 是否合理
- 拉取当前登录用户名下全部 Bug 并做产品判断
- 结合需求文档、版本文档、PRD、验收口径分析 Bug
- 基于开发、测试、产品在 Bug 中的回复给出处理建议

## 输入约定

优先识别以下输入：

- `bug_ids`: 指定一个或多个 Bug 号
- `assigned_to_me`: 是否拉取当前登录用户名下全部 Bug
- `doc_paths`: 本地需求文档路径，支持一个或多个
- `zentao_doc_ids`: 禅道文档 ID，支持一个或多个
- `version_keyword`: 版本关键词，如 `575`、`V3398`
- `product_keyword`: 产品关键词，用于辅助缩小分析范围

如果用户没有提供需求文档位置，先追问：

- 本地文档路径
- 禅道文档 ID
- 是否允许先做“不结合需求文档”的初步判断

## 执行流程

### 1. 检查禅道会话

1. 先调用 `checkZentaoSession`
2. 如果未登录或 session 无效，调用 `zentaoLogin`
3. 登录失败时，停止分析并明确告知用户

### 2. 获取 Bug 数据

根据用户意图选择其一：

- 指定 Bug 号：对每个 Bug 调用 `getZentaoBugDetail` 和 `getZentaoBugComments`
- 拉取我名下全部 Bug：先调用 `getZentaoUserBugs`

对于“我名下全部 Bug”：

- 如果数量较少，直接逐条补充详情与评论
- 如果数量很多，按批次分析并输出统一汇总
- 不要只给标题列表，必须补齐详情和评论后再判断

必要时可结合 `searchZentaoBugs` 进一步按版本、产品、状态筛选。

### 3. 获取需求文档

按用户提供的位置读取文档：

- 本地文档：直接读取文件内容
- 禅道文档：调用 `getZentaoDocDetail`

如果文档内容引用附件、截图或文件路径，可用 `getZentaoFileContent` 获取可访问链接，但仅在附件确实影响判断时再使用。

如果用户提供了多个文档，优先按以下方式匹配 Bug：

1. 版本关键词
2. 模块/功能名称
3. Bug 标题与描述中的业务对象

若仍无法准确匹配，明确标注“文档匹配存在不确定性”。

### 4. 分析维度

每个 Bug 至少从以下角度判断：

1. `问题是否清晰`
   - 标题是否准确
   - 描述、步骤、期望结果、实际结果是否足够
2. `是否符合需求文档`
   - 文档是否明确支持测试结论
   - 是否属于需求未写清、规则冲突或验收口径缺失
3. `各方回复是否改变判断`
   - 开发回复是否说明实现限制、历史逻辑或非缺陷原因
   - 测试回复是否补充了复现路径、环境、边界条件
   - 产品回复是否给出了业务规则或最终口径
4. `问题归类`
   - 合理缺陷
   - 需求遗漏/文档不清
   - 非缺陷但需优化体验
   - 使用方式或测试口径偏差
   - 信息不足，暂时无法判断

### 5. 输出规则

先给结论，再给依据，再给建议。避免只复述 Bug 内容。

单个 Bug 使用以下模板：

```markdown
## Bug {id} - {title}

- 结论：合理 / 部分合理 / 存疑 / 暂无法判断
- 归类：代码缺陷 / 需求遗漏 / 文档不清 / 非缺陷但建议优化 / 信息不足
- 关键依据：
  - Bug 标题与描述中的核心问题
  - 开发/测试/产品回复中的关键信号
  - 需求文档中的对应规则
- 建议动作：
  - 对产品：是否补充 PRD、补充验收口径、统一业务规则、与测试/开发同步
  - 对测试：是否补充复现条件、截图、账号、数据前置条件
  - 对开发：是否需要按缺陷修复、按优化处理、或先等待产品口径
```

批量分析时，先输出总览，再输出逐条结果：

```markdown
# 禅道 Bug 分析结果

## 总览
- 分析范围：指定 Bug / 我名下全部 Bug
- Bug 总数：N
- 合理缺陷：X
- 需求遗漏或文档不清：Y
- 非缺陷但建议优化：Z
- 信息不足：M

## 优先处理建议
- 先补需求口径的项
- 先与测试确认复现条件的项
- 先进入开发修复的项

## 逐条分析
```

## 判断原则

- 需求文档明确支持测试结论时，优先判为“合理缺陷”
- 需求文档与当前实现一致、但测试诉求偏向体验优化时，优先判为“非缺陷但建议优化”
- 需求文档缺失或表述模糊时，不要强行定性为代码缺陷，优先归类为“需求遗漏/文档不清”
- 若 Bug 描述、评论和文档互相冲突，要把冲突点单独列出
- 若证据不足，不要猜测，直接说明需要补充的信息

## 追问规则

在以下情况必须先追问再继续：

- 未提供 Bug 号，且也未明确要拉取“我名下全部 Bug”
- 需要结合需求文档，但用户未提供文档路径或文档 ID
- Bug 数量过大，且用户还希望按特定版本、模块或状态过滤

推荐追问项：

- 需求文档在本地哪个路径，还是禅道文档 ID？
- 是分析指定 Bug 号，还是分析当前登录用户名下全部 Bug？
- 是否只看某个版本、某个产品、某个状态？

## 示例触发语

- `分析禅道 Bug 12345，需求文档在 docs/prd/v575.md`
- `拉取我名下全部 Bug，结合禅道文档 4926 分析合理性`
- `分析 12345、12346、12380，这三个是 V575 的问题，需求文档在 docs/prd/v575.md`
- `把我名下 active 的 Bug 做一轮产品判断，文档路径是 docs/需求/V576-结算改版.md`
