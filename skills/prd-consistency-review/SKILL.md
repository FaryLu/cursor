# PRD Consistency Review

## 前置上下文
执行本 skill 前，必须读取：@skills/_shared/prd-context.md
- @skills/refine-prd-detail/SKILL.md

## 作用定位
本 skill 用于在用户提供“需求描述 + 指定 PRD 文档”后，审核 PRD 是否能进入评审或研发对接。

重点检查：
- 功能逻辑是否完整
- 与历史功能文档、历史版本、接口或现有模块规则是否冲突
- 字段、按钮、权限、数据范围是否符合 `refine-prd-detail` 的颗粒度要求
- 文档上下文是否自洽
- 是否存在错别字、歧义、模糊表述、容易误解的规则

本 skill 默认只做审核和修订建议，不直接重写整篇 PRD。若用户明确要求“帮我改成最终版”，可在审核后输出可替换文案。

## 输入要求
用户输入至少应包含：
- 需求描述
- 指定产品需求文档路径或文档正文
- 需求类型：`现有功能迭代` / `优化` / `新功能`

若需求类型缺失，先根据描述尝试判断；无法判断时必须追问。

若用户标明是现有功能迭代或优化，且要求对照历史逻辑，用户应提供需要读取的本地历史功能文档路径。若未提供，但无法仅凭本地搜索定位到明确历史文档，必须追问。

## 资料读取顺序
必须按以下顺序执行，不允许跳步：

1. 读取用户指定的 PRD 文档。
2. 读取需求描述，提取需求类型、模块、页面、字段、关键业务动作、上下游单据、权限点、版本号。
3. 读取 `refine-prd-detail`，作为字段、按钮、权限、数据范围颗粒度检查标准。
4. 按需求类型读取历史资料：
   - `现有功能迭代` 或 `优化`：优先读取用户指定的本地历史功能文档；再按关键词补充搜索本地历史 PRD。
   - `新功能`：必须搜索本地历史 PRD、MCP 中 `user-mzfileserver` 的历史版本记录；如已配置 swagger MCP 或用户提供 swagger / 接口文档，也要搜索接口或字段定义。
5. 对照 `秒账功能关联关系.md` 与 `秒账核心功能.md` 检查是否遗漏关联模块、横切能力和兼容项。
6. 输出审核结论、问题清单、冲突说明和建议修订文案。

## 本地历史检索
默认检索范围：
- `../../data/prd/*/*.md`
- `../../data/modules/秒账核心功能.md`
- `../../data/modules/秒账功能关联关系.md`
- 用户明确指定的历史功能文档
- 用户明确指定的 swagger / 接口 / 字段定义文件

检索关键词必须至少覆盖：
- 需求标题和核心对象
- 主模块、子模块、页面名称
- 关键字段、按钮、状态、权限点
- 上游单据、下游单据、报表、打印、导入、导出、业务日志、老数据兼容

## MCP 检索规则
调用任何 MCP 工具前，必须先读取对应工具 descriptor，确认参数后再调用。

### user-mzfileserver
新功能必须查询；现有功能迭代或优化在本地历史文档不足时也必须查询。

优先使用：
- `search_by_keyword`：按需求关键词、字段、业务动作搜索。
- `search_by_module`：按模块名搜索同模块历史需求。
- `search_by_version`：当需求描述或历史文档出现版本号时搜索版本。
- `get_requirement_detail`：当搜索结果中出现可疑相似需求时读取详情。

### swagger
若当前环境存在 swagger MCP、swagger 文件或接口文档，按以下方式处理：
- 搜索与本次 PRD 涉及模块、字段、状态、枚举、接口入参、接口出参相关的定义。
- 检查 PRD 字段名、字段含义、枚举值、状态流转与接口定义是否一致。
- 若未配置 swagger MCP 且用户未提供接口文档，不得编造接口结论，只能在输出中标记“接口/Swagger 未检查”。

## 检查维度

### 1. 功能逻辑完整性
检查 PRD 是否说明：
- 入口、页面、区域、字段、按钮、状态、异常提示
- 新建、编辑、查看、删除、导入、导出、打印、审核、反审核等相关动作
- 必填、默认值、取值来源、格式、长度、联动、校验、报错提示
- 上下游单据联动、回写、反查、状态变化
- 老数据兼容、新账号默认值、商户设置开关、货品属性、偏好设置

### 2. 历史版本冲突
对照历史功能文档、本地历史 PRD、MCP 记录和 swagger，检查：
- 同一字段名称是否含义变化
- 同一状态是否流转规则冲突
- 同一按钮是否权限或点击逻辑变化
- 新规则是否破坏老数据、老流程、老用户习惯
- 历史文档中已有明确规则时，当前 PRD 是否说明复用、覆盖或兼容方式

冲突必须按“当前 PRD 写法 -> 历史依据 -> 冲突原因 -> 建议处理”输出。

### 3. 字段、模块和权限颗粒度
必须按 `refine-prd-detail` 的标准检查：
- 列表字段是否先区分基础字段、敏感字段、条件字段、计算字段、选择型字段
- 字段权限、操作权限、数据权限是否混写
- 搜索区、列表区、详情页、新建页、编辑页是否分别写清楚
- 下拉框、选择器、联想搜索是否说明候选数据来源和数据范围
- 金额、成本、均价、售价、毛利、利润等敏感字段是否写清楚权限表现
- 按钮是否写清楚显示条件、操作权限、点击逻辑、前置校验、成功失败反馈

### 4. 上下文自洽性
检查：
- 前后名称、字段、模块、页面、状态是否一致
- 需求背景、影响模块、功能描述、验收标准是否互相支撑
- 同一规则是否在不同章节出现不同口径
- 是否出现“前文说支持，后文又排除”的矛盾
- 是否有未定义名词、未解释缩写、未落地的泛化描述

### 5. 文案质量
检查：
- 错别字、漏字、多字
- 容易引起研发、测试、业务方理解偏差的句子
- “同现有”“按原逻辑”“适配一下”“相关数据”“正常展示”等无依据表达
- 无法验收的描述，例如“尽量”“合理”“优化体验”

## 优先级规则
- P0：会导致功能做错、历史逻辑冲突、数据错误、权限风险、库存/金额/状态异常、无法验收。
- P1：会导致研发或测试理解偏差、遗漏边界条件、遗漏兼容方案、字段或按钮规则不完整。
- P2：文案表达、结构顺序、术语一致性、可读性优化。

## 必须追问条件
出现以下任一情况，先追问，不要强行审核：
- 未提供 PRD 文档且无法从上下文定位。
- 需求类型无法判断，且会影响历史检索路径。
- 现有功能迭代或优化，但用户要求对照指定历史文档却没有提供文档路径。
- PRD 涉及金额、库存、状态流转、权限、上下游单据，但需求描述缺少关键规则且历史资料无法补足。
- 用户要求检查 swagger，但当前环境没有 swagger MCP、swagger 文件或接口文档。
